# End-to-End Schema-to-SQL Pipeline

A compound AI system that transforms natural language business requirements into normalized relational database schemas (DDL), and autonomously synthesizes corresponding valid SQL queries.

## 🏗️ Architecture Overview

The system is built as a **two-phase pipeline** using two fine-tuned LLMs working in sequence. By decoupling schema design from query synthesis, we achieve highly accurate and structurally sound SQL outputs without hallucinating non-existent columns.

### Phase 1: Natural Language → DDL (Schema Design)
* **Input:** A plain-English description of the database requirements (e.g., *"An e-commerce app with users, orders, and products"*).
* **Model:** Fine-tuned LLM.
* **Output:** A mathematically precise SQL `CREATE TABLE` DDL statement, containing all tables, columns, primary keys, and foreign key relationships.

### Phase 2: DDL → SQL (Query Synthesis)
* **Input:** The DDL schema generated by Phase 1.
* **Model:** Qwen2.5-Coder-1.5B (Fine-tuned using LoRA).
* **Output:** Diverse, compilable `SELECT` queries that operate exclusively on the provided schema.

## 🛠️ Data Engineering & Preprocessing (`model_2/preprocess.py`)

Rather than relying on raw SQL strings, we process the Spider Dataset programmatically from its source `tables.json` file. This enables advanced mathematical transformations:

1. **Table-Boundary Truncation:** LLMs have limited token context windows (e.g., 1024 tokens). Truncating a long schema blindly will slice queries mid-table (e.g., `CREATE TABL...`), breaking syntax. Our algorithm calculates token lengths iteratively, adding whole tables until the budget is exhausted, ensuring perfectly well-formed schemas.
2. **Name-Masking Augmentation:** Models often "cheat" by using English semantic clues (e.g., assuming a table named `users` joins with `user_email`). We mathematically replace real table/column names with generic placeholders (`table_0`, `col_0`). This forces the model to learn the mathematical graph structure (Foreign Keys) instead of English memorization.
3. **Substring Collision Prevention:** When masking names, replacing `user` before `user_profile` would corrupt the latter into `table_0_profile`. Our pipeline sorts regex replacements in descending length order (`key=lambda x: -len(x[0])`) to prevent partial-match collisions.

## 🚀 Training & Fine-Tuning (`model_2/train_qwen.py`)

Training a 1.5 Billion parameter model from scratch would cause OOM (Out Of Memory) crashes and lead to *Catastrophic Forgetting* of its pre-trained coding knowledge. 

We utilize **Parameter-Efficient Fine-Tuning (PEFT)** via **LoRA**:
* **Rank (r=16):** Compresses a 1024x1024 matrix update into `1024x16 * 16x1024`, slashing trainable parameters from 1.5 Billion to just ~10 Million.
* **Target Modules:** We inject LoRA adapters directly into the Attention modules (`q_proj`, `k_proj`, `v_proj`, `o_proj`, `gate_proj`, `up_proj`, `down_proj`).
* **Batching:** We use a physical `batch_size=1` to fit in VRAM, but use `gradient_accumulation_steps=8` to mathematically simulate a larger, stable batch size of 8.

---

## 🧪 Ephemeral Evaluation Framework (`model_2/evaluate_qwen.py`)

Traditional exact-string-matching for SQL evaluation is flawed (e.g., `SELECT a, b` is functionally identical to `SELECT b, a`, but string matching marks it wrong). 

Instead, we employ an **Ephemeral Execution Framework**:
* **Diverse Sampling:** We use `do_sample=True` and a higher `temperature` to flatten the probability curve. This forces the model to explore token branches and generate structurally different query variations (Pass@K metrics).
* **Live SQLite Compilation:** We connect to the actual Spider `.sqlite` databases and run `EXPLAIN QUERY PLAN` on the generated output. If it executes without a syntax or schema error, it is structurally valid.
* **Diversity & Uniqueness Scoring:** The evaluation script actively scans outputs for SQL constructs (`JOIN`, `GROUP BY`, `HAVING`) to ensure the model isn't just generating simple `SELECT *` queries.

## 💻 Running the Demo

To run the interactive End-to-End Live Pipeline:

```bash
# Ensure you are at the repository root
python true_pipeline_demo.py
```

1. The script will prompt you for a natural language description.
2. Phase 1 will generate the DDL schema.
3. Phase 2 will autonomously generate 5 diverse, valid SQL queries based on that schema.
4. Python's Garbage Collector (`gc.collect()`) and `torch.cuda.empty_cache()` are used actively between phases to ensure the 1.5B models can safely cycle without crashing VRAM.
